# Правила работы агентов

## Обязательный контекст

Перед любыми изменениями агент обязан прочитать:

1. `docs/nauchny_klubok_top1_tz.md` — основной план работ, продуктовые требования, MVP, top-1 ориентиры и правила для ИИ-агентов.
2. Этот файл `AGENTS.md` — единые правила репозитория, git-процесса и коммуникации.
3. `docs/agent_prompts/system.md` — системный промпт строгой работы по ТЗ.
4. `docs/agent_prompts/before_implementation.md` — чек перед имплементацией.
5. `docs/agent_prompts/quality_gate.md` — финальная проверка качества.
6. `docs/agent_context/project_structure.md` — актуальная карта структуры проекта.
7. `docs/agent_context/sync_rules.md` — правила синхронизации контекста между агентами.

Если агентная система не подхватывает контекст автоматически, пользователь или агент должен явно вставить в новый чат:

```text
Работаем в репозитории ScientificTangle. Сначала прочитай `AGENTS.md`, `docs/nauchny_klubok_top1_tz.md`, `docs/agent_prompts/system.md` и `docs/agent_context/project_structure.md`, затем выполняй только текущую задачу. README не трогай без отдельного явного запроса.
```

## Общие правила разработки

- Главный ориентир проекта — `docs/nauchny_klubok_top1_tz.md`; решения не должны противоречить MVP, архитектурным ограничениям и demo-сценарию из ТЗ.
- Перед имплементацией сначала обдумай архитектурное место изменения, план, риски и критерии проверки.
- Делай минимально достаточные изменения под текущую задачу, без переписывания соседнего кода и без добавления лишних фич.
- Не меняй `README.md` без отдельного явного запроса.
- Не хардкодь demo-ответы, факты без источников, контракты без синхронизации и security-правила без отдельного решения команды.
- Перед завершением задачи проверь, что изменения можно объяснить через Definition of Done из ТЗ или через явный запрос пользователя.

## Код и документация

- В коде не должно быть комментариев.
- Исключения для комментариев допустимы только если они обязательны для внешнего формата, лицензии, генератора, миграции, линтера или юридического требования.
- Вместо комментариев улучшай имена, структуру кода и русскоязычную Markdown-документацию.
- Документация проекта ведется на русском языке автоматически, если пользователь явно не попросил другой язык.
- При добавлении новой значимой директории, сервиса, модуля, контракта, схемы, миграции, пайплайна или инфраструктурного файла обновляй `docs/agent_context/project_structure.md` в том же коммите.
- Если меняется правило, которое должны знать все агенты, синхронно обновляй `AGENTS.md`, релевантные адаптеры агентов и файлы в `docs/agent_prompts/` или `docs/agent_context/`.

## Git, ветки и коммиты

- Рабочие ветки проекта: только `dev` и `feat/*`.
- Все агентные изменения пушатся только в `dev` или в `feat/*` для последующего вливания в `dev`.
- Агенты не пушат в `main`, не делают force-push в `main` и не мержат в `main`.
- Успешные изменения из `dev` в `main` переносит только человек вручную.
- Перед любым push, PR или merge создай проверочную копию текущей ветки или целевой ветки, чтобы исходная ветка оставалась безопасной точкой отката.
- Для проверки `feat/*` сначала создай копию `dev`, влей туда `feat/*`, прогони проверки и только после стабильного результата переноси изменения в `dev`.
- Если агент находится в `main`, он должен переключиться на `dev` или создать `feat/*` до внесения изменений.
- Формат коммита: `feat: сделано то-то`.
- Коммит всегда в одну строку, на русском языке, без scope.
- Перед коммитом проверь `git status` и не добавляй чужие/локальные файлы случайно.
- В коммитах, push/PR-описаниях и публичной истории не упоминай Cursor, Codex, Claude, Antigravity, ZCode или другую агентную систему; описывай только результат работы.

## Коммуникация с пользователем

- Пиши кратко, по делу, на русском, если пользователь не попросил иначе.
- Если задача явно стала другой задачей, мягко предложи новый чат:

```text
Это уже отдельная задача. Для более эффективной работы лучше начать новый чат и вставить туда такой промпт:
Работаем в репозитории ScientificTangle. Сначала прочитай `AGENTS.md`, `docs/nauchny_klubok_top1_tz.md`, `docs/agent_prompts/system.md` и `docs/agent_context/project_structure.md`. Нужно сделать: <кратко описать новую задачу>.
```

- Если контекст перегружен или агент начинает терять связность, предложи новый чат с тем же промптом и кратким summary текущего состояния.
- Не навязывай процесс, но явно предупреждай о рисках: потеря контекста, случайное изменение `README.md`, push в `main`, merge без проверочной копии ветки, коммит не по формату, рассинхрон агентных правил.
