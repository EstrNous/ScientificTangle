# Правила умной синхронизации агентов

## Цель

Все агентные системы должны видеть один и тот же контекст: ТЗ, правила разработки, структуру проекта, промпты, git-процесс и ограничения качества.

## Обязательные файлы общего контекста

- `AGENTS.md`
- `docs/nauchny_klubok_top1_tz.md`
- `docs/agent_prompts/system.md`
- `docs/agent_prompts/before_implementation.md`
- `docs/agent_prompts/quality_gate.md`
- `docs/agent_context/project_structure.md`
- `docs/agent_context/sync_rules.md`

## Правило синхронного обновления

Если меняется одно из правил, которое должно быть видно всем агентам, агент обязан обновить:

- `AGENTS.md` как главный источник правил;
- адаптеры конкретных систем, если им нужна короткая ссылка на новое правило;
- `docs/agent_prompts/*`, если правило влияет на поведение при решении задач;
- `docs/agent_context/project_structure.md`, если меняется структура проекта.

## Git-синхронизация

- Рабочие ветки проекта: только `dev` и `feat/*`.
- Перед началом задачи агент обязан стянуть актуальный `dev`: `git fetch origin dev` и `git pull --ff-only origin dev` в локальной ветке `dev`.
- Новые `feat/*` ветки создаются только от актуального локального `dev` после успешного fast-forward обновления.
- Все изменения агентов пушатся в `dev` или в `feat/*` с последующим вливанием в `dev`.
- `main` не является рабочей веткой для агентов.
- Агенты не пушат в `main`, не делают force-push в `main` и не мержат в `main`.
- Успешный перенос из `dev` в `main` делает только человек вручную.
- Перед push, PR или merge создается проверочная копия ветки.
- Для проверки `feat/*` создается копия `dev`, в нее вливается `feat/*`, затем выполняются проверки; только стабильный результат переносится в `dev`.
- В коммитах, push/PR-описаниях и публичной истории не упоминаются Cursor, Codex, Claude, Antigravity, ZCode или другие агентные системы.

## Документация

Документация проекта ведется на русском языке автоматически, если пользователь явно не попросил другой язык.

При добавлении нового модуля, сервиса, контракта, схемы, пайплайна или инфраструктурного элемента агент должен добавить краткое русскоязычное описание в релевантный общий файл контекста.

## Код

В коде не должно быть комментариев.

Исключения возможны только если комментарий обязателен для внешнего формата, лицензии, генератора, миграции, линтера или юридического требования. В обычном прикладном коде комментарии не добавлять; вместо комментариев улучшать имена, структуру и документацию на русском в Markdown-файлах.
